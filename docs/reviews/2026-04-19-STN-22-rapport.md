## Ce qui m'a surpris

- **TopBar livré en stub mais avec a11y déjà câblée** (`aria-expanded`/`aria-controls`/`aria-label`) — c'est plus avancé qu'un scaffold typique. Bonne base, évite la dette a11y classique.
- **Le `<button>` overlay** pour fermer la Sidebar — choix sémantiquement discutable (un overlay cliquable n'est pas un bouton au sens WAI-ARIA) mais pragmatique pour obtenir le focus + clavier gratuit. Mérite d'être revisité à l'audit a11y.
- **Aucun commit `refactor: blue`** dans l'historique TDD. Légitime ici (rien à refactorer à partir d'une implé propre), mais vaut la peine de noter que Blue reste optionnel quand Green est déjà idiomatique.

## Décisions à revisiter

- **Flex vs Grid** : le DOR STN-22 mentionnait "Grid CSS Tailwind", l'implémentation utilise flex-col+flex. Équivalent fonctionnel ici, mais grid aurait été plus robuste si un footer fixe apparaît plus tard. Décision pragmatique à confirmer avec l'équipe en rétro.
- **Sidebar stub avec un seul item "Accueil"** : acceptable en dev, mais risque d'être mergé en prod tel quel si STN-23 traîne. À cadrer : condition d'affichage (cacher le stub si 1 seule route), ou merger STN-23 avant release v0.1.0.

## Patterns à surveiller

**À répliquer** :
- Séparation `useSidebarToggle` (état) / `Sidebar` (présentation) / `TopBar` (contrôleur) — pattern propre à reprendre pour d'autres composants UI stateful (dialog, toast, etc.).
- `data-open={isOpen}` avec sélecteur Tailwind `data-[open=true]:...` — élégant, évite les `className` conditionnels spaghetti.

**À surveiller / éviter** :
- `<button>` utilisé comme overlay sémantique — valide techniquement mais crée une confusion sémantique. Si le pattern se répand, factoriser un composant `Overlay` explicite.

## Questions ouvertes

- Jusqu'où va la responsabilité de STN-22 en termes d'a11y ? Le DOR n'exige pas de focus trap, mais l'équipe cyber (audit STN-XX à venir) va potentiellement remonter ces points. **À clarifier en rétro** : intégrer un item "a11y minimum shipable" dans la DOR générique frontend.
- **Convention de commit Blue** : faut-il rendre explicite qu'un `refactor: blue —` peut être absent quand Green est déjà propre ? Mettre à jour le skill `/tdd` pour clarifier.
- **`feat` vs `refactor` pour les améliorations post-review** : le commit `1187509` ajoute Link + Escape + close-on-navigate. Taggé `feat`. Pourrait aussi être lu comme "phase Blue tardive". À clarifier dans `/tdd`.

- **Déstructuration pour stabilité des deps useEffect** — bonne façon de répondre à `react-hooks/exhaustive-deps` (plutôt que `eslint-disable`). À citer quand quelqu'un rencontre ce warning.
- **Leçon "suggestion défensive qui casse"** — l'ajout du guard `if (isOpen) close()` semblait plus propre mais a introduit un bug (l'effet re-déclenchait au toggle, annulant l'ouverture). Rappel : ajouter une variable aux deps d'un useEffect change la fréquence d'exécution, pas juste le contenu. À citer en rétro / onboarding.
- **Commit `877962c` = exemple du bon réflexe "fix le code, pas le test"** — quand le test E2E a échoué (click Link vers même route), l'ajout d'`onClick={onDismiss}` sur le Link rend le **comportement réel** plus robuste, au lieu d'ajouter une 2e route juste pour faire passer le test. À citer en onboarding comme pattern "tests révèlent des gaps produit".
