## Décisions à revisiter

- **Placement `app/email/` vs `app/core/email/`** — corrigé dans cette PR après challenge. **Règle à valider en équipe** : *toute chose ayant un domaine métier → feature slice top-level, `core/` réservé à la plumbing transverse sans entités*.
- **Profile `smtp` pour MailHog** — choix pragmatique (pas démarré par défaut). À reconsidérer si on fait souvent tourner `docker compose up` et qu'on veut MailHog always-on en dev.
- **`SMTP_USE_STARTTLS=false` par défaut dans `.env.example`** — sécurise pour MailHog dev mais pourrait être un piège si quelqu'un copie `.env.example` en prod sans surcharger. STN-35 (SOPS) doit **imposer** `true` pour les envs preview/prod.

## Patterns à surveiller

**À répliquer** :
- **Log sans fuite** (`subject_hash`) — généralisable à tout log manipulant des données sensibles (mots de passe hash, tokens, identifiants).
- **Pyramide tests unit (mocks) + integ (testcontainers)** — pattern clair et rapide. À systématiser pour chaque adapter infrastructure.
- **`try/except sur l'infra uniquement`** (comme documenté dans le docstring SmtpEmailSender). Bon exemple pour onboarding.

**À surveiller / éviter** :
- `finally: await resource.quit()` **sans** try/except → peut masquer l'exception en cours. Pattern récurrent en Python async, à flagger dans les reviews.
- Paramètre `cause` d'exception qui duplique `raise ... from` → redondance Python facile à commettre.

## Questions ouvertes

- **Faut-il standardiser un helper `async_safe_close(resource)` dans `app/shared/`** ? Vu qu'on va avoir plusieurs adapters async (DB, Redis, SMTP, LLM), le pattern `try: await resource.close() except: pass` dans des `finally` va se multiplier.
- **Politique retry sur `EmailDeliveryException`** — hors scope STN-34, mais à trancher avant la 1re use case qui l'utilise (magic link auth). Retry au niveau use case ou au niveau infrastructure (wrapper) ?
