## Ce qui m'a surpris

- **`react-router-dom@7` déjà installé** (pas v6 comme le DOR le précise) — v7 est rétro-compatible sur l'API `createBrowserRouter`, donc aucun blocage technique, mais le DOR mérite d'être mis à jour (ou un ticket de doc dédié pour acter le choix v7).
- **`core/router.tsx` existait déjà** (livré par STN-10 ou STN-22) avec juste HomePage sur `/`. J'ai étendu plutôt que recréé — respect du principe "pas de pré-création" implicite.
- **Tension règle "pas de pré-création de dossiers features" vs ce ticket** — le DOR demande 6 pages placeholder (Login, Dashboard, Catalog, Deployments, Chat, NotFound) pour les features auth/dashboard/catalog/deployment/chat. J'ai résolu en les mettant dans **`core/pages/`** plutôt que dans les dossiers de features — cohérent avec la règle (les features naissent avec leur ticket, les placeholders vivent dans core en attendant).
- **eslint `react-refresh/only-export-components`** a forcé à splitter `AuthProvider.tsx` en 3 fichiers (`AuthContext.ts` + `AuthProvider.tsx` + `useAuth.ts`). Pattern "1 export par concern" bien appliqué, même si verbeux.

Non livré dans ce ticket (scope réduit aux tests unit/integ). *À ajouter dans un ticket `test(STN-XX): E2E smoke navigation` une fois le frontend a plus de contenu (post-charte graphique).*

## Décisions à revisiter

- **`/` redirige vers `/dashboard`** — choix pragmatique, mais en non-authentifié, ça déclenche `/dashboard` → ProtectedRoute → redirect `/login`. Double redirect. Alternative : avoir `/` rendre directement `/login` si non auth, `/dashboard` si auth. Moins déclaratif mais évite le double redirect. **À challenger quand le vrai auth arrive.**
- **`AppLayout` appliqué à `/` et enfants mais pas à `/login` et `/404`** — choix : login et 404 sont full-screen sans TopBar/Sidebar. Standard UX. Mais mérite validation visuelle par Yassine quand la charte sera livrée (STN-154).
- **Utilisation de la fonction helper `protect()`** — réduit la verbosité mais pourrait surprendre (un reviewer pourrait préférer l'inline `<ProtectedRoute>{...}</ProtectedRoute>`). À conserver si l'équipe préfère la lisibilité du router.

## Patterns à surveiller

- **`core/pages/` pour placeholders** vs `<feature>/pages/` pour les pages implémentées — à généraliser comme convention. Documenter dans CLAUDE.md si validé.
- **Tests integ router via `createMemoryRouter` + fixture `renderAt()`** — pattern réutilisable pour tous les futurs tests de navigation. À garder.
- **Split provider + context + hook en 3 fichiers** (react-refresh/only-export-components) — à appliquer pour tout nouveau Context React.

## Questions ouvertes

- **v6 → v7 dans CLAUDE.md** : mettre à jour le stack documenté ?
- **`core/pages/` est-ce acceptable à long terme** ou anti-pattern (devrait tout être migré vers les feature folders dès que possible) ?
- **Niveau de protection de `/login`** : quand l'utilisateur est déjà authentifié et va sur `/login`, faut-il le rediriger vers `/dashboard` (pattern "inverse ProtectedRoute") ? Non bloquant pour v0.1 mais UX concernée.

## Points positifs notables

- **Tests `it.each()` pour les 4 routes protégées** — DRY efficace, bon pattern pour éviter la duplication de tests similaires.
- **Non-régression complète** : 48/48 tests passent (10 existants + 12 nouveaux), 0 warning lint, 0 erreur typecheck, format OK.
- **Convention de nommage FR respectée** dans les tests (`describe`/`it` en français).
- **AuthProvider exposable avec `value` optionnel** — permet de tester sans wrapper supplémentaire, et de fournir un faux user en dev si besoin.
