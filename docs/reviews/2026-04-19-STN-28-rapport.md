## Ce qui m'a surpris

- **`prevent_destroy` ne supporte pas les expressions conditionnelles en Terraform** — c'est une limitation peu documentée qui a forcé un compromis (safe-by-default sur tous les envs au lieu de prod uniquement). Mérite d'être capturée dans la doc d'équipe.
- **STN-12 a mergé pendant la review de STN-28** et a apporté `docker-compose.yml` + `apps/api/app/health/` sans conflit (scopes disjoints). Bonne démonstration que le vertical slicing + découpage fin des tickets fonctionne en équipe.
- Le `.terraform.lock.hcl` est versionné par env (4 fichiers). Aucune automatisation pour garantir qu'ils restent synchronisés → drift silencieux possible si un dev lance `init -upgrade` dans un seul env.

## Dette technique identifiée

- [x] ~~**Test `prevent_destroy` par grep statique**~~ — **fixé dans cette PR** : parsing Python du module (regex multiline) qui vérifie qu'**exactement chaque** `docker_volume` a son propre bloc `lifecycle { prevent_destroy = true }`. Catche le cas "je déplace le prevent_destroy sur une autre ressource".
- [x] ~~**Whitelist `*.tfvars` sans protection**~~ — **fixé dans cette PR** : nouveau script `tests/infra/test_tfvars_no_secrets.sh` qui scanne les patterns sensibles (`password`, `secret`, `token`, `api_key`, `aws_*_access`, etc.). Appelable en CI + futur pre-commit hook. Le vrai hook husky reste à faire dans un ticket dédié (config du repo).
- [x] ~~**Pas de validation que `env_name` correspond au dossier**~~ — **fixé dans cette PR** : chaque `environments/<env>/main.tf` a un bloc `validation { condition = var.env_name == "<env>" }` avec message d'erreur explicite.
- [ ] **Drift potentiel entre `.terraform.lock.hcl` des 4 envs** — pas d'automatisation pour garder en sync. *À surveiller — pas encore bloquant, vérifier lors de la migration backend remote (v0.2+).*
- [ ] **Hook pre-commit husky/lint-staged** qui invoque `test_tfvars_no_secrets.sh` automatiquement avant chaque commit. *Ticket à créer quand la config husky globale sera en place. ~1 SP.*

## Décisions à revisiter

- **`prevent_destroy = true` sur tous les envs** (pas uniquement prod) — choix pragmatique dicté par la limitation Terraform. Impact concret sur le dev loop : un `terraform destroy` en dev échouera, obligeant à passer par `terraform state rm`. À rediscuter en équipe si la friction devient un point de douleur quotidien. Alternative : accepter la duplication de 2 blocs `docker_volume` par env avec `count` conditionnel.
- **Doc `destroy` propose `terraform state rm`** comme workflow de suppression — à cacher derrière une section ⚠️ "exceptionnel, pas routine" pour éviter d'ancrer un mauvais réflexe.

## Patterns à surveiller

- **Pattern `dynamic labels` dans le module `env`** — à généraliser aux futures ressources Docker (container, volume, network) pour garder les labels standards cohérents à l'échelle du projet.
- **Convention de nommage `stacknest-<env>-*`** bien posée — à faire respecter par tous les tickets infra suivants (STN-29 Nginx, STN-14 env.sh).
- **Script de test `test_*.sh` avec fallback `terraform → terraform.exe`** — bon pattern cross-OS, à réutiliser pour tout script shell exécuté sous Windows (WSL ou Git Bash).

## Questions ouvertes

- L'équipe est-elle alignée sur le compromis `prevent_destroy` global ? À valider en prochaine synchro dev.
- Faut-il migrer vers un backend Terraform remote (S3/Azure) **avant** d'aller en prod ? Actuellement state local → perte de state = perte de traçabilité. Sujet à trancher dans un ticket dédié v0.2+.
- Quand passer les env-specific tfvars sous SOPS ? STN-31 est identifiée mais pas priorisée — à positionner avant la 1ère feature qui aura besoin d'un secret (SMTP, BDD credentials, etc.).
